THIRD-PARTY NOTICES
===================

This file lists third-party code vendored into the provenance-neovim-recorder
(provnvim) source tree. A Neovim plugin is distributed as its own source tree,
so anything vendored here is redistributed by us. Each entry records the
upstream project, the vendored path in this repo, the version/commit it was
taken from, and the full upstream license + copyright text.

TweetNaCl.js (ed25519 signatures + SHA-512)
-------------------------------------------

Vendored path : lua/provenance/vendor/ed25519.lua
Upstream       : TweetNaCl.js — https://github.com/dchest/tweetnacl-js
Version        : 1.0.3
Commit         : e6141a20553498697e9e7d92fbc4517994caff52
Source file    : nacl.js (the non-"fast" reference build)
License        : The Unlicense (public domain dedication)

What was taken and how it was adapted
  Ported to LuaJIT (Neovim's bundled runtime): the ed25519 signature scheme
  (crypto_sign / crypto_sign_open / crypto_sign_keypair, the GF(2^255-19) field
  arithmetic, and the Edwards-curve group operations) together with the SHA-512
  implementation TweetNaCl bundles (crypto_hash / crypto_hashblocks), which
  ed25519 requires internally per RFC 8032. Adaptations from the JavaScript
  source: Uint8Array/Float64Array became 0-indexed Lua tables of numbers (LuaJIT
  numbers are IEEE-754 doubles, matching JS); `subarray` views became explicit
  integer offset parameters; the SHA-512 64-bit words use a {hi,lo} table of
  unsigned 32-bit halves with the LuaJIT `bit` library for the bitwise ops;
  JavaScript int32 operators (^ & | ~ << >>>) map to LuaJIT `bit`
  (bxor/band/bor/bnot/lshift/rshift), while arithmetic `>>` and `Math.floor` on
  wide field values map to `math.floor` division. The Salsa20/Poly1305/X25519
  parts of TweetNaCl.js were NOT vendored. Correctness is proven byte-for-byte
  against the pinned cross-language vector in tests/conformance/fixtures/
  ed25519.json (produced by @noble/ed25519 in the Provenance monorepo).

PLC — Pure Lua Crypto (XChaCha20-Poly1305 AEAD)
-----------------------------------------------

Vendored path : lua/provenance/vendor/xchacha20poly1305.lua
Upstream       : philanc/plc (Pure Lua Crypto) — https://github.com/philanc/plc
Version        : commit 9efed4113a1a5dcb12a1f526cd65561756e6d648
Source files   : plc/chacha20.lua, plc/poly1305.lua, plc/aead_chacha_poly.lua
License        : MIT

What was taken and how it was adapted
  The ChaCha20 stream cipher (RFC 8439), the HChaCha20 subkey derivation and
  XChaCha20 24-byte-nonce variant, the Poly1305 MAC (poly1305-donna 32-bit
  limb schedule), and the RFC 8439 §2.8 AEAD combiner (16-byte tag appended,
  empty AAD supported) were ported into a single module. The Salsa20, ec25519,
  SHA2, and other primitives in plc were NOT vendored.

  Adaptations from the plc source, which targets Lua 5.3+ (native 64-bit
  integers, the & ~ << >> | bitwise operators, and string.pack/string.unpack):
    * Neovim's runtime is LuaJIT with Lua 5.1 semantics — numbers are IEEE-754
      doubles, there is no string.pack, and bitwise work goes through the `bit`
      library. ChaCha20's 32-bit word operations were rewritten with `bit`
      (bxor/rol/band/rshift) using bit.tobit for mod-2^32 addition, and all
      little-endian packing/unpacking is done by hand.
    * Poly1305's poly1305-donna 32-bit reduction accumulates products/sums that
      reach ~2^58, relying on Lua 5.3's 64-bit integers. On LuaJIT doubles
      (53-bit mantissa) that would lose precision, so the d0..d4 multiply-
      accumulate is performed in an explicit 64-bit {hi,lo} representation
      (a mul64 helper plus carry propagation) before the >>26 reduction. The
      26-bit limb schedule and the modular reduction are otherwise plc's,
      unchanged.
  Correctness is proven byte-for-byte against RFC 8439 (§2.4.2 ChaCha20,
  §2.5.2 Poly1305), draft-irtf-cfrg-xchacha-03 §A.3.1 (XChaCha20-Poly1305),
  and the pinned cross-language vector in tests/conformance/fixtures/
  session-key.json (produced by @noble/ciphers in the Provenance monorepo).
